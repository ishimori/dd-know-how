# 開発フローのステップ数調整

プロジェクトの規模やチームの状況に合わせて、開発フローのステップ数を調整できます。

## フローのバリエーション

| バリエーション | ステップ数 | 推奨対象 |
|---------------|-----------|---------|
| ミニマル | 3ステップ | 個人開発、プロトタイプ |
| スタンダード | 5ステップ | 小〜中規模チーム |
| 詳細 | 7ステップ | 中〜大規模チーム |
| フル | 9ステップ | 大規模、品質重視 |

---

## 3ステップ（ミニマル）

最小限のフローです。

```
┌─────────────────────────────────────────┐
│  Step 1: DD作成 + 仕様確認              │
│  Step 2: 実装 + テスト + 検証            │
│  Step 3: 完了（コミット + アーカイブ）    │
└─────────────────────────────────────────┘
```

### 各ステップの内容

| Step | 内容 |
|------|------|
| 1 | DDを作成し、何を作るか明確化 |
| 2 | コーディング、テスト、動作確認まで一気に |
| 3 | コミットしてDDをアーカイブ |

### 進捗表示

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 Step N/3: ステップ名
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 1. DD作成       ▶️ 2. 実装         ⬜ 3. 完了
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 適した状況

- 個人開発
- プロトタイプ作成
- 小さな修正・改善

---

## 5ステップ（スタンダード）

バランスの取れた標準的なフローです。

```
┌─────────────────────────────────────────┐
│  Step 1: DD作成                         │
│  Step 2: 仕様確認                       │
│  Step 3: 実装                           │
│  Step 4: 検証（テスト + レビュー）       │
│  Step 5: 完了（コミット + アーカイブ）    │
└─────────────────────────────────────────┘
```

### 各ステップの内容

| Step | 内容 |
|------|------|
| 1 | DDを作成し、目的・タスクを整理 |
| 2 | 仕様を確認し、認識合わせ |
| 3 | コーディング + テスト作成 |
| 4 | 動作確認 + コードレビュー |
| 5 | コミットしてDDをアーカイブ |

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 Step N/5: ステップ名
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 1. DD作成       ✅ 2. 仕様確認     ▶️ 3. 実装
⬜ 4. 検証         ⬜ 5. 完了
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 適した状況

- 小〜中規模チーム
- 週次〜月次のリリースサイクル
- 品質とスピードのバランスを重視

```
┌─────────────────────────────────────────┐
│  Step 1: DD作成                         │
│  Step 2: 仕様確認                       │

│  Step 4: テスト作成                     │
│  Step 5: コード検証（Lint + 動作確認）   │
│  Step 6: レビュー                       │
│  Step 7: 完了（コミット + アーカイブ）    │
└─────────────────────────────────────────┘
```

### 各ステップの内容

| Step | 内容 |
|------|------|
| 1 | DDを作成 |
| 2 | 仕様を確認し、認識合わせ |
| 3 | コーディング |
| 4 | テストを作成 |
| 5 | Lint実行 + 動作確認 |
| 6 | コードレビュー |
| 7 | コミットしてDDをアーカイブ |

## 9ステップ（フル）

仕様書連携を含む完全なフローです。

```
┌─────────────────────────────────────────┐
│  Step 1: DD作成                         │
│  Step 2: 仕様確認                       │
│  Step 3: 実装前チェック                  │
│  Step 4: コーディング                    │
│  Step 5: テスト作成                     │
│  Step 6: コード検証                     │
│  Step 7: レビュー                       │
│  Step 8: 仕様書同期                     │
│  Step 9: コミット + アーカイブ           │
└─────────────────────────────────────────┘
```

### 各ステップの内容

| Step | 内容 |
|------|------|
| 1 | DDを作成 |
| 2 | 仕様を確認し、認識合わせ |
| 3 | 仕様書・既存コードを事前確認 |
| 4 | コーディング |
| 5 | テストを作成 |
| 6 | Lint実行 + セルフチェック + 動作確認 |
| 7 | コードレビュー |
| 8 | DDの変更を仕様書に反映 |
| 9 | コミットしてDDをアーカイブ |

### 進捗表示

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📍 Step N/9: ステップ名
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 1. DD作成       ✅ 2. 仕様確認     ✅ 3. 実装前チェック
▶️ 4. コーディング  ⬜ 5. テスト作成   ⬜ 6. コード検証
⬜ 7. レビュー      ⬜ 8. 仕様書同期   ⬜ 9. コミット
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 適した状況

- 大規模チーム
- 仕様書の整備が重要なプロジェクト
- 品質を最優先するプロジェクト
- 監査対応が必要なプロジェクト

## フローの選び方

### 判断基準

| 基準 | 3ステップ | 5ステップ | 7ステップ | 9ステップ |
|------|-----------|-----------|-----------|-----------|
| チーム規模 | 1人 | 2-5人 | 5-10人 | 10人以上 |
| リリース頻度 | 随時 | 週次 | 隔週 | 月次以下 |
| テスト重視度 | 低 | 中 | 高 | 最高 |
| 仕様書の重要度 | 低 | 中 | 中 | 高 |
| 品質要件 | 低 | 中 | 高 | 最高 |

### 移行パス

```
3ステップ → 5ステップ → 7ステップ → 9ステップ
```

プロジェクトの成長に合わせて段階的にステップを増やすことをお勧めします。

## カスタムフロー

上記のバリエーション以外にも、プロジェクトに合わせてカスタマイズできます。

### 例: 6ステップ（レビュー重視）

```
DD作成 → 仕様確認 → 実装 → レビュー → 修正 → 完了
```

### 例: 4ステップ（テスト重視）

```
DD作成 → 実装 → テスト → 完了
```

重要なのは、チームで合意したフローを一貫して使うことです。

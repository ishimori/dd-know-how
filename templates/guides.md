# DD作成ガイド

## 1. アプローチ選択

ユーザーが指定すればそれに従う。指定がなければ自動判断し、DD冒頭に選択理由を1行明示する。

| 優先順 | 判断基準 | アプローチ | 差分テンプレート |
|:------:|---------|-----------|----------------|
| 1 | 画面の新規作成・大幅変更 | モック先行 | `dd_template_mock.md` |
| 2 | 計算・遷移・ビジネスロジックが中心 | TDD | `dd_template_tdd.md` |
| 3 | 上記以外 | 標準（探索的実装） | `dd_template.md` のみ |

記載例: `> アプローチ: モック先行（画面新規作成のため）`

Phaseごとにアプローチを混在させてもよい（例: BE=TDD、FE=モック先行）。AskQuestion はスコープ自体が不明瞭な場合の最終手段。

## 2. テンプレート適用

`dd_template.md` を骨格とし、差分テンプレートがあれば該当Phaseを差し替える。

## 3. 命名規則

```
DD-{番号}_{タイトル}.md       # 標準（番号は3桁ゼロパディング）
DD-083-1_子タイトル.md        # 親子DD（同テーマの続き）
```

## 4. フォルダ構成

```
doc/DD/
├── DD-001_タイトル.md    # DD本体（常に直下）
├── DD-001/               # 添付フォルダ（あれば）
└── archived/             # 完了したDD（添付フォルダごと移動）
```

**❌ 禁止:** `attachment/` や `attachments/` 等の汎用的な名前のフォルダを作成しないこと。

番号管理: 新規作成時は `doc/DD/` と `archived/` の両方から最大番号を確認。

## 5. DD本体のサイズ管理

**200行以内。** 超えそうなら:
- **添付ファイル分離**: 50行超の仕様・エビデンス・モック・シナリオ → `DD-{番号}/` に配置
- **子DD分割**: 別テーマとして切り出せる部分 → 子DDに

### 粒度＝レビューゲートの単位

**1つのDDは、1回のレビューサイクル（作業→成果物→ユーザーレビュー→合意）で完結する単位にする。**

レビューゲートが1回 → DD単体で完結（親子にしない）:
```
DD-070: ログ出力フォーマット変更（実装→レビュー→完了）
```

レビューゲートが2回以上 → 親DD + 子DDに分割:
```
DD-050: XX機能開発（親。管理用。実装は子DDで行う）
DD-050-1: DB設計（ER図レビュー→合意）
DD-050-2: 画面モック（HTMLレビュー→合意）
DD-050-3: FE/BE実装（動作確認→合意）
```


## 6. タスク記述ルール

- 各タスクに対象ファイルパスを明記する
- 変更内容は具体的に書く（「更新」「改修」だけは禁止）
- 各Phaseに🔬機械検証タスクを入れる（実行コマンド + 期待結果）

## 7. テスト戦略

テストコードを書く前に、必ず自然言語でシナリオを作成しユーザーの合意を得ること。詳細は差分テンプレート（`dd_template_tdd.md` / `dd_template_mock.md`）を参照。

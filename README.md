# DD-Know-How

**LLMの楽観バイアスを制御し、設計判断を記録し続ける開発フレームワーク**

Claude Code のスキルとして動作する、DD（Design Document）ドリブンの開発管理システムです。

---

## 解決する問題

AIコーディングアシスタントには構造的な弱点があります。

| 問題 | 結果 |
|------|------|
| **楽観バイアス** | 「問題ありません」で見落としを素通りする |
| **コンテキスト忘却** | セッションをまたぐと設計判断の経緯が消える |
| **場当たり実装** | 全体像なしに目の前のコードだけ書く |
| **自己レビューの甘さ** | 自分が書いたコードを「正しい」前提で確認する |
| **ゴミテスト量産** | 意味のないテストコードを大量に書く |

DD-Know-How はこれらに対して、仕組みで対抗します。

## DDとは何か

DDは「意思決定の記録」「タスク管理」「作業ログ」を1つのMarkdownファイルにまとめたものです。

```markdown
# DD-042: 認証方式の変更

## 目的        ← 何を、なぜやるのか
## 検討内容    ← どんな選択肢があったか
## 決定事項    ← 最終的に何を選んだか
## タスク一覧  ← Phase分割された作業リスト（各PhaseにDA批判レビュー付き）
## ログ        ← 時系列の作業記録
## DA批判レビュー記録  ← 各Phaseで「何が壊れるか」を探した記録
```

DDはセッションをまたいでも残り続けます。3ヶ月後に「なぜこの設計にしたのか」を振り返れる。これが、コンテキストが揮発するLLMとの開発で最も価値を持つ部分です。

## 3つのアプローチ

DD作成時、タスクの性質に応じてアプローチを選択します。

| アプローチ | 用途 | テンプレート |
|-----------|------|-------------|
| **標準（探索的実装）** | 既存パターンの踏襲、設計判断が少ないタスク | `dd_template.md` |
| **TDD** | 計算・ビジネスロジックが中心で「正解」が明確 | `dd_template.md` + `dd_template_tdd.md`（差分） |
| **モック先行** | 画面の新規作成・大幅変更 | `dd_template.md` + `dd_template_mock.md`（差分） |

ユーザーが指定しなければ、CLAUDEが自動判断してDD冒頭に選択理由を明示します。詳細は `templates/guides.md` を参照。

### DDの粒度＝レビューゲートの単位

**1つのDDは、1回のレビューサイクルで完結する単位にする。**

- レビューゲートが1回 → DD単体で完結
- レビューゲートが2回以上 → 親DD + 子DDに分割

```
DD-050: XX機能開発（親。管理用）
DD-050-1: DB設計（ER図レビュー→合意）
DD-050-2: 画面モック（HTMLレビュー→合意）
DD-050-3: FE/BE実装（動作確認→合意）
```

DD本体は **200行以内**。超えそうなら添付ファイル分離か子DD分割で対応。

## 核心: DA批判レビュー（Devil's Advocate）

最大の特徴は **DA批判レビュー** です。各実装フェーズの完了前に、LLMに対して批判的視点を強制します。

```
❌「確認したか？」 → 「はい、問題ありません」
⭕「どこが壊れるか？」 → 壊れる前提で探すしかない
```

### 4段階の批判手順

1. **壊れる前提で探す** — 「完了した」ではなく「何が壊れるか」と問う
2. **暗黙の前提を疑う** — 「今動いている」ではなく「前提が崩れるケースは？」
3. **将来の破壊を予測** — 「今は問題ない」ではなく「半年後の変更で壊れないか？」
4. **発見を記録** — 何も見つからないのは疑わしい。変更ファイルを読み直して再チェック

### DA品質フィルター（ノイズ削減）

| # | ルール | 内容 |
|---|--------|------|
| 1 | **検証義務** | 指摘前にコード実行 or 仕様確認で事実確認。推測だけの指摘は記録しない |
| 2 | **既知事項の除外** | `TODO`/`FIXME`、「Phase X で対応」、別DDスコープアウト済みは指摘対象外 |
| 3 | **深刻度の判定基準** | 高=今のPhaseで壊れる、中=特定条件下で壊れる、低=改善提案 |
| 4 | **再現手順の必須化** | 高/中には「何をすると→何が起きるか」を必ず記載 |

詳細は [doc/da-method.md](doc/da-method.md) を参照。

## テスト戦略: シナリオレビューゲート

LLMに任せると意味のないテストコードを量産する問題への対策。

**テストコードを書く前に、必ず自然言語でシナリオを作成しユーザーの合意を得る。** コードを先に書かない。E2Eは「人間がブラウザから手入力するのと同じこと」を再現する。

## 機械検証: LLMの「やったふり」防止

LLMは「確認しました」と報告しながら実際にはコマンドを実行していないことがある。各Phaseに機械検証タスク（コマンド実行 + 期待結果の照合）を入れることで、エビデンスを強制する。

```markdown
- [ ] `shared/types.ts` の SiteRole 型を 'admin' → 'site_admin' に変更
- [ ] 🔬 `npx tsc --noEmit` で型エラーなしを確認
```

## 導入方法

### 自分のプロジェクトに導入する

```bash
cd dd-know-how
claude
> /setup /path/to/your-project
```

2段階の導入レベルから選択:

| レベル | 内容 |
|--------|------|
| Level 1 | テンプレート + ガイド。まず試したい場合 |
| Level 2 | + `/dd` スキル + DA メソッド文書。通常はこれで十分 |

手動セットアップの詳細は [IMPORT.md](IMPORT.md) を参照。

### DD-Know-How 自体を使う

```bash
git clone https://github.com/ishimori/dd-know-how.git
cd dd-know-how
claude
> /dd new ログイン機能の実装
```

## 実プロジェクトでの運用実績

2つのフルスタックプロジェクトで DD-Know-How を継続運用した実績があります。

### プロジェクトA（React + Firebase）

| 指標 | 数値 |
|------|------|
| DD総数 | 80件（進行中10件 + アーカイブ70件） |
| プロジェクト規模 | 画面27枚、Firestore 10+コレクション、外部API 3種連携 |

### プロジェクトB（Next.js + FastAPI + PostgreSQL）

| 指標 | 数値 |
|------|------|
| DD総数 | 94件 |
| プロジェクト規模 | 画面10枚+、PostgreSQL、さくらVPSデプロイ |

プロジェクトBでの運用を通じて、アプローチ選択（標準/TDD/モック先行）、レビューゲート単位の粒度管理、シナリオレビューゲート等の知見を得てフレームワークに反映しています。

### DA批判レビューの実効性

DA批判レビューが実際にバグを発見した例（匿名化）:

**例1: 関連ファイルの同種バグ発見**
```
Phase 2 のDA批判レビューで、修正対象ファイル以外にも
同じハードコード値が使われていることを発見。
→ 関連3ファイルを追加修正、テスト16件追加
```

**例2: 非機能要件の達成確認**
```
各サブDD完了時のDAレビューで、ビルドサイズ目標（300KB gzip以下）を
毎回計測・記録。逸脱を早期に検知する仕組みとして機能。
→ 最終ビルド: 221.96 KB gzip（目標達成）
```

詳細な実例は [doc/examples/](doc/examples/README.md) を参照。

## 設計思想

### なぜ「見落としチェック」ではなくDevil's Advocateか

初期バージョンでは「何を見落としているか？」と問う形式でした。しかし実運用で判明したのは、LLMは「見落とし」を問われると「特にありません」と答えがちだということです。

「どこが壊れるか？」と問い方を変えるだけで、LLMは壊れるポイントを探すモードに切り替わります。

### なぜ差分テンプレートか

TDD・モック先行は標準テンプレートのPhase構成を差し替えるだけです。完全独立の3テンプレートにすると共通部分の同期コストが発生するため、差分方式を採用しています。差分方式でLLMの合成精度に問題が出た場合は、独立テンプレートに切り替える判断も可能です。

## 複数人での運用

識別子プレフィックスで番号衝突を防ぎます。

```
DDI-001, DDI-002   # 石森の番号系列
DDSA-001, DDSA-002 # 斉藤の番号系列
```

# DD-Know-How

**LLM支援開発のための、設計記録 × 批判レビュー × 機械検証 の方法論**

Claude Code のスキルとして動作する、DD（Design Document）ドリブンの開発管理システムです。実プロジェクト運用から抽出した経験則を方法論として体系化しています。

---

## このリポジトリは何を解いているか

AIコーディングアシスタント（Claude Code等）には構造的な弱点があります。

| 問題 | 結果 | このリポジトリの対策 |
|------|------|----------|
| **楽観バイアス** | 「問題ありません」で見落としを素通り | **DA批判レビュー** |
| **やったふり** | 「確認しました」と報告しつつ未実行 | **機械検証タスク** |
| **コンテキスト忘却** | セッションをまたぐと設計判断が消える | **DD（設計文書）** |
| **場当たり実装** | 全体像なしに目の前のコードだけ書く | **Phase 0設計判断** |
| **批判レビューの過剰化** | 網羅性のために水増し指摘 → Critical閾値低下 | **DA品質フィルター** |
| **ゴミテスト量産** | 意味のないテストコードを大量生成 | **シナリオレビューゲート** |

このリポジトリの中核は、これらに対する **DD・DA・機械検証** の3本柱です。

---

## 3本柱の関係

```
┌────────────────────────────────────────────────────┐
│                                                    │
│   DD（設計記録）    ── 何を、なぜ、どう作ったか    │
│        ↓                                           │
│   DA（批判レビュー）── 何が壊れるか                │
│        ↓                                           │
│   機械検証          ── 嘘がつけないエビデンス      │
│                                                    │
└────────────────────────────────────────────────────┘
```

- **DDだけ**ではコンテキスト記録に留まり、品質は担保されない
- **DAだけ**では論理的批判で終わり、実行時の証拠が残らない
- **機械検証だけ**では合否は分かるが「なぜ」が残らない

3つを組み合わせて初めて、LLM開発の信頼性が方法論として成立します。

---

## 柱1: DD（Design Document）

DDは「意思決定の記録」「タスク管理」「作業ログ」を1つのMarkdownファイルにまとめたものです。

```markdown
# DD-042: 認証方式の変更

## 目的        ← 何を、なぜやるのか
## 検討内容    ← どんな選択肢があったか
## 決定事項    ← 最終的に何を選んだか
## タスク一覧  ← Phase分割された作業リスト（各PhaseにDA批判レビュー付き）
## ログ        ← 時系列の作業記録
## DA批判レビュー記録  ← 各Phaseで「何が壊れるか」を探した記録
```

### DDが解決する2つの問題

**問題1: 設計判断の揮発**
セッションをまたぐとLLMはコンテキストを失います。DDに「なぜこの設計を選んだか」を書いておくと、3ヶ月後の自分や他のメンバー（人間/AI問わず）が振り返れます。

**問題2: ナレッジ転記の手作業**
従来は「DD完了時にMEMORY.mdに転記する」というコンテキスト回収作業が必要でした。**DDはファイルパス・設計判断・gotchaを全て本体に記録するため、`grep` で全文検索できるナレッジベースとして機能します。** 転記不要です。

### 5つのアプローチ

DD作成時、タスクの性質に応じてアプローチを選択します。

| 優先順 | アプローチ | 用途 |
|:------:|-----------|------|
| 1 | **バグ修正** | 既知のバグ修正（再現→修正→Before/After検証） |
| 2 | **モック先行** | 画面の新規作成・大幅変更（見た目の合意が必要） |
| 3 | **E2E駆動** | 画面の振る舞い中心（操作→結果の検証が必要） |
| 4 | **TDD** | 計算・遷移・ビジネスロジックが中心 |
| 5 | **標準（探索的実装）** | 上記以外 |

ユーザーが指定しなければCLAUDEが自動判断し、DD冒頭に選択理由を明示します。Phaseごとにアプローチを混在させることも可能（例: BE=TDD、FE=モック先行）。詳細は [templates/guides.md](templates/guides.md) 参照。

### DDの粒度 = レビューゲートの単位

**1つのDDは、1回のレビューサイクルで完結する単位にする。** レビューゲートが2回以上必要なら親DD + 子DDに分割。DD本体は **200行以内**。

## 柱2: DA（Devil's Advocate）批判レビュー

最大の特徴は **DA批判レビュー** です。各実装Phaseで、LLMに対して批判的視点を強制します。

```
❌「確認したか？」 → 「はい、問題ありません」
⭕「どこが壊れるか？」 → 壊れる前提で探すしかない
```

問い方を変えるだけで、LLMは「見落としを探す」モードに切り替わります。

### 実運用から抽出された経験則

DAは単なるチェックリストではなく、実プロジェクト運用から得られた以下の経験則を持ちます。

#### Phase 0 DA（実装前）が最も価値が高い

ある機能開発（主菜A/B選択制）の初期設計をPhase 0でDA分析した結果、Critical 5件を発見し**全て実装前に修正**。各Phaseでの手戻りはゼロでした。

> 30分〜1時間のPhase 0 DAで、数時間〜数日の手戻りが防げる

#### 7ラウンドDA: Round 2でピークが来る

別案件（プロンプト生成一本化）で同じコードベースに角度を変えてDAを繰り返した結果:

```
Round 1: ■             方針の妥当性
Round 2: ■■■■■         3システム全差分比較（ピーク）
Round 3: ■■■           ランタイム・インポートチェーン
Round 4: ■■■           内部契約・既存バグ
Round 5: ■■■           循環import・DOM依存
Round 6: ■■■           sections契約・純粋関数性
Round 7: ◯             収束確認
```

初回は浅い探索、**2回目で深い比較分析に切り替えた時にピーク**を迎え、以降は漸減する。1回のDAで満足してはいけない。

#### 修正後DAで「同根パターン」を潰す

`state.bannedGroups = []` を `state.bannedGroups.length = 0` に修正した後のDAで、**同じパターンが別の2箇所にも存在**することを発見。バグ修正後は同根パターンのDA必須。

#### DAの副産物として無関係バグが見つかる

DD-003のDA（プロンプト統合の設計検証）の過程で、**対象DDとは無関係な既存バグ5件**を発見。DAは「対象DDの設計検証」に限定せず、触れたコードの品質全般を見る。

#### DA 0件が妥当なケースもある

UIリファクタリング（DOM構造の整理のみ）でCritical 0件で完了した例。ロジック変更がなく既存テストが安全性を担保している場合、Phase内DAだけで十分。**全てのDDに深掘りDAが必要なわけではない。**

### DA品質フィルター: 「件数」より「読み返す価値」

DAには対称的な落とし穴があります。網羅性のために水増しすると、**Critical判定の閾値が相対的に下がり、本当のCriticalが埋もれる**のです。実運用では、初回DAで出た指摘の半数以上が「実装に着手すれば自然に気付くレベル」だったケースが頻発します。

#### 4つのフィルター

| # | ルール | 内容 |
|---|--------|------|
| 1 | **検証義務** | 指摘前にコード実行 or 仕様確認で事実確認。推測だけの指摘は記録しない |
| 2 | **既知事項の除外** | `TODO`/`FIXME`、「Phase X で対応」、別DDスコープアウト済みは指摘対象外 |
| 3 | **再現手順の必須化** | 高/中には「何をすると→何が起きるか」を必ず記載 |
| 4 | **「読み返す価値」基準** | 後から読み返した時に判断の根拠として使える粒度に限る |

#### 「読み返す価値」基準（フィルター4の詳細）

指摘は以下のいずれかに該当するもののみ:

1. **次Phase着手後に確実に詰まる地雷** — 仕様・スキーマ・型・前提条件を読み直して初めて気付くレベルの非互換・欠落
2. **スコープ判断（やる/やらない）の合意が要る項目** — 本DDの境界を決める判断で、合意なしで進むと後で衝突する
3. **過去の失敗の再発予防に効く運用ルール** — 同種の事故を繰り返さないための仕掛け

**対象外**（Phase実装時の自然な配慮で吸収できる）:
- 命名・型変換・引数検証など、コードを書き始めれば気付く事項
- 「次Phaseで実測すれば足りる」レベルのメモ
- 実害のない軽微な不整合

> 品質指標は**件数ではなく「読み返した時に判断の根拠として使えるか」**

詳細・実例は [doc/da-method.md](doc/da-method.md) を参照。

## 柱3: 機械検証（やったふり防止）

LLMは「確認しました」と報告しながら実際にはコマンドを実行していないことがあります。これに対する直接的な対策が機械検証です。

**機械検証 = 人間が判断しなくてもコマンド出力で合否が分かる検証**

```markdown
- [ ] `shared/types.ts` の SiteRole 型を 'admin' → 'site_admin' に変更
- [ ] 🔬 `npx tsc --noEmit` で型エラーなしを確認
- [ ] 🔬 `npm test` で全テストパスを確認
```

期待結果まで書くことで、LLMは「テスト確認しました」と曖昧に答える余地がなくなります。各Phaseに最低1つの機械検証タスクを入れることが原則です。

機械検証の典型例:
- テスト実行（`npm test`, `pytest`）
- 型チェック（`tsc --noEmit`, `mypy`）
- ビルド（`npm run build`）
- Lint（`eslint`, `ruff`）
- 整合性チェック（grep / 自作チェッカー）

目視が必要なもの（UXレビュー等）とは明示的に区別します。

## 補強する仕組み

### コーディング基準書

[templates/coding-standards.md](templates/coding-standards.md) に P01〜P22 の原則を定めた基準書を同梱しています。**YES/NOの二値で判定できる形式**で記述されており、コードレビュー時の評価基準として機能します。

- **カテゴリ A〜G**: 型安全性、コンポーネント設計、非同期処理、API通信、レイヤー分離、バリデーション、DB操作
- **カテゴリ H**: LLM固有のアンチパターン（型システム迂回、スタブ残置、デバッグコード、SQLインジェクション）
- **採点**: S/A/B/C の4段階。Critical違反が1件でもあればC

### シナリオレビューゲート（テスト戦略）

LLMに任せるとゴミテストを量産する問題への対策。**テストコードを書く前に、必ず自然言語でシナリオを作成しユーザーの合意を得る。** コードを先に書かない。E2Eは「人間がブラウザから手入力するのと同じこと」を再現する。

### スクリーンショットエビデンス

画面を伴うPhase完了時、Playwright MCP の `browser_evaluate` でCSSを注入し赤枠ハイライトしてキャプチャ。バグ修正はBefore/Afterペア、新規開発は追加箇所ハイライト。詳細は [templates/guides.md](templates/guides.md) セクション8。

## 導入方法

### 自分のプロジェクトに導入する

```bash
cd dd-know-how
claude
> /setup /path/to/your-project
```

2段階の導入レベルから選択:

| レベル | 内容 |
|--------|------|
| Level 1 | テンプレート + ガイド。まず試したい場合 |
| Level 2 | + `/dd` スキル + DAメソッド文書。通常はこれで十分 |

手動セットアップの詳細は [IMPORT.md](IMPORT.md) を参照。

### DD-Know-How 自体を使う

```bash
git clone https://github.com/ishimori/dd-know-how.git
cd dd-know-how
claude
> /dd new ログイン機能の実装
```

## 設計思想

### なぜ「見落としチェック」ではなくDevil's Advocateか

初期バージョンでは「何を見落としているか？」と問う形式でした。しかし実運用で判明したのは、LLMは「見落とし」を問われると「特にありません」と答えがちだということです。**「どこが壊れるか？」と問い方を変えるだけで、LLMは壊れるポイントを探すモードに切り替わります。**

### なぜ差分テンプレートか

5つのアプローチは、標準テンプレート（`dd_template.md`）を骨格とし、アプローチ固有のPhaseだけを差分テンプレートで差し替える方式です。

```
dd_template.md（共通骨格: 目的・検討・決定・タスク・ログ）
  + dd_template_bugfix.md → バグ修正のPhase構成に差し替え
  + dd_template_tdd.md    → TDDのPhase構成に差し替え
  + dd_template_mock.md   → モック先行のPhase構成に差し替え
  + dd_template_e2e.md    → E2E駆動のPhase構成に差し替え
  （差分なし）             → 標準（探索的実装）
```

完全独立の4テンプレートにすると共通部分の同期コストが発生するため、差分方式を採用。差分方式でLLMの合成精度に問題が出た場合は、独立テンプレートに切り替える判断も可能です。

### DAの限界も明示する

DAは万能ではありません。偽陽性が出る、ラウンドを重ねると重複発見が増える、実行時の問題（タイミング依存・負荷）は見つけにくい、レビュアー（LLM）のドメイン知識に依存する、といった限界があります。**DAで「0件」になっても「見つけられなかった」可能性を常に意識し、テスト・E2E・実ユーザーフィードバックと組み合わせて信頼性を補完します。**

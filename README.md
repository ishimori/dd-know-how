# DD-Know-How

**AI支援開発に「設計の記録」と「疑う工程」を組み込む — Claude Code 用の仕様駆動開発キット**

Claude Code は速い。しかし放っておくと「なぜそう作ったか」はセッションと共に消え、「動きました」という報告は検証されないまま積み上がります。DD-Know-How は、実プロジェクトでのDD運用（数百件規模）から抽出した方法論を、Claude Code の **スキル・テンプレート・フック・スクリプト** としてそのまま導入できる形にパッケージしたものです。

こんな経験があるなら、このリポジトリはあなたのためのものです:

- 3ヶ月前にAIと書いたコード、**なぜこの設計なのか誰も説明できない**
- 「テスト通りました」を信じたら、**実はコマンドを実行していなかった**
- レビューを頼むと「問題ありません」。**リリース後に壊れる**
- 仕様書が実装に追いつかず、**正しい仕様を知る手段が grep しかない**

## 使うとこうなる

導入後の Claude Code との典型的なやり取りです（要約）:

```text
あなた  > /dd new 会員一覧のCSVエクスポート

Claude  > アプローチ: 標準（探索的実装）を選択（ロジック中心のため）
          doc/DD/DD-013_会員一覧のCSVエクスポート.md を作成しました。

          ## 受け入れ基準
          | # | 基準（操作 → 期待結果）                              | 検証方法        |
          | 1 | 「CSV出力」→ 表示中の絞り込み条件で出力される          | Phase 2 E2E #1 |
          | 2 | 10万件でもタイムアウトしない                          | Phase 1 機械検証 |

          Phase 0（事前精査 + DA批判レビュー）から始めます。

Claude  > 😈 Phase 0 DA批判レビュー
          [高] 論理削除済み会員がエクスポートに混入する
               再現: 削除済み会員がいる状態で全件出力 → CSVに含まれる
               → 決定事項に「deleted_at IS NULL を必須条件にする」を追記
          [中] 文字コード未定義（Excel は BOM なし UTF-8 を文字化けさせる）
               → 受け入れ基準 #3 として追加

          （…実装。各Phaseの最後に必ず:）

Claude  > 🔬 機械検証: npm test → 42 passed / npx tsc --noEmit → エラーなし
          📸 エビデンス: DD-013/csv-export-after.png（追加ボタンを赤枠ハイライト）

Claude  > 🤖 別モデル・クロスレビュー（別モデル＝Codexに差分を委譲・任意）: Phase 0で「推奨」と判定
          codex-review.sh --check → 利用可 / 差分を Codex CLI にヘッドレスで委譲
          [中] 大量件数の出力でメモリにCSV全体を保持している（ストリーミング未対応）
               → 受け入れ基準 #2（10万件でタイムアウトしない）と抵触 → 修正タスク追加

あなた  > /dd archive 13

Claude  > 仕様書同期チェック: doc/spec/A030_会員一覧.md に CSV出力の記述なし → 追記
          知見の昇格判定: 「Excel は BOM 付き UTF-8 が必要」→ engineering-patterns.md に昇格
          DD-INDEX.md 再生成 → doc/archived/DD/ へ移動完了
```

ポイントは3つ。**(1) 完成条件が最初に文書化される**、**(2) AIが自分の設計を疑う工程が必ず入る（さらに別モデルにも監査させられる）**、**(3) 報告にはコマンド出力とスクリーンショットの証拠が付く**。そして完了時には、仕様書と知見が自動的に「現在形」へ反映されます。

## クイックスタート（3分）

前提: [Claude Code](https://claude.com/claude-code)（CLI / VSCode拡張 / デスクトップ）。テンプレート・ドキュメントは日本語です。

```bash
git clone https://github.com/ishimori/dd-know-how.git
cd dd-know-how
claude
> /setup /path/to/your-project
```

対話形式で導入レベルを選びます:

| レベル | 内容 | 推奨 |
|--------|------|------|
| Level 1（最小） | テンプレート + ドキュメント体系（DOC-MAP / spec / 知見ファイル） | まず試したい |
| **Level 2（標準）** | + `/dd` スキル + DAメソッド + フック + スクリプト | **通常はこちら** |

セットアップ時に **Playwright MCP**（AIに「目」を与え、スクリーンショットエビデンスを自動化）と **Lintヒント基盤**（後述）の導入も提案されます。手動導入の手順は [IMPORT.md](IMPORT.md) を参照。

導入が終わったら:

```bash
> /dd new はじめての機能
```

## 何を解いているか

AIコーディングアシスタントには構造的な弱点があります。それぞれに具体的な対策を当てています。

| 問題 | 結果 | 対策 |
|------|------|------|
| **楽観バイアス** | 「問題ありません」で見落としを素通り | **DA批判レビュー** |
| **やったふり** | 「確認しました」と報告しつつ未実行 | **機械検証 + エビデンス** |
| **コンテキスト忘却** | セッションをまたぐと設計判断が消える | **DD（設計文書）** |
| **完成条件の曖昧さ** | 「できました」の基準がAI任せ | **受け入れ基準** |
| **場当たり実装** | 全体像なしに目の前のコードだけ書く | **Phase 0 設計判断** |
| **仕様書の風化** | 実装だけ進み正典が古くなる | **spec同期チェック + doc-check** |
| **知見の揮発** | 同じ罠に何度もはまる | **engineering-patterns / decisions への昇格** |
| **批判レビューの水増し** | 網羅性のための指摘で Critical が埋もれる | **DA品質フィルター** |
| **同一モデルの盲点** | 実装したモデルの自己レビューでは同じ見落としを見逃す | **別モデル・クロスレビュー（任意）** |
| **ゴミテスト量産** | 意味のないテストコードを大量生成 | **シナリオレビューゲート** |

## 中核: 3本柱

```
   DD（設計記録）    ── 何を、なぜ、どう作ったか
        ↓
   DA（批判レビュー）── どこが壊れるか
        ↓
   機械検証          ── 嘘がつけないエビデンス
```

DDだけでは品質が担保されず、DAだけでは証拠が残らず、機械検証だけでは「なぜ」が残らない。3つの組み合わせで初めて、AI開発の信頼性が方法論として成立します。

### 柱1: DD（Design Document）

意思決定・受け入れ基準・タスク・作業ログを1つのMarkdownにまとめた、**1レビューサイクル = 1ファイル** の設計文書です。

```markdown
# DD-042: 認証方式の変更
## 目的            ← 何を、なぜやるのか
## 検討内容        ← どんな選択肢があったか
## 決定事項        ← 最終的に何を選んだか
## 受け入れ基準    ← 何ができたら完成か（操作 → 期待結果、検証方法と1対1対応）
## タスク一覧      ← Phase分割（各Phaseに機械検証 + DA批判レビュー付き）
## ログ            ← 時系列の作業記録
## DA批判レビュー記録
```

DD本体にファイルパス・設計判断・gotcha がすべて残るため、**grep で全文検索できるナレッジベース**として機能します（`/dd search キーワード`）。タスクの性質に応じて5アプローチ（バグ修正 / モック先行 / E2E駆動 / TDD / 標準）を自動選択し、差分テンプレートでPhase構成を切り替えます。

### 柱2: DA（Devil's Advocate）批判レビュー — 「疑う工程」の土台

「疑う工程」は2層です。**① 同一セッション内の自己批判（DA）** が常時動く土台で、**② 別モデルによるクロスレビュー**（次節）が同一モデルの死角を突く最終層。まず①から。

```
❌「確認したか？」   → 「はい、問題ありません」
⭕「どこが壊れるか？」→ 壊れる前提で探すしかない
```

問い方を変えるだけで、AIは「見落としを探すモード」に切り替わります。実運用から抽出した経験則ごと同梱しています:

- **Phase 0（実装前）のDAが最も価値が高い** — ある機能の初期設計で Critical 5件を実装前に発見・修正し、手戻りゼロ。30分のDAが数日の手戻りを防ぐ
- **1回で満足しない** — 同じコードに角度を変えて7ラウンド実施した実例では、2回目（差分比較に切り替えた時）に発見のピークが来た
- **修正後は「同根パターン」を探す** — 1箇所直したバグと同じパターンが、別の2箇所から見つかった実例
- **0件が妥当なケースもある** — ロジック変更のないUIリファクタリング等。件数ノルマは置かない

そして対称的な落とし穴への対策が **DA品質フィルター**: 検証義務（推測だけの指摘は記録しない）・既知事項の除外・再現手順の必須化・**「読み返す価値」基準**（実装すれば自然に気付くレベルの指摘で件数を稼がない）。品質指標は件数ではなく「後から判断の根拠として使えるか」です。詳細と実例は [doc/da-method.md](doc/da-method.md)。

**そして②が、同じモデルの自己レビューでは越えられない壁への答えです。** DAをどれだけ回しても、実装がClaude系なら残るのはClaude系の死角。そこで実装差分を *実装したのとは別のモデル* に監査させる — それが2層目の **別モデル・クロスレビュー** です（次節で詳述）。

### 柱3: 機械検証（やったふり防止）

**機械検証 = 人間が判断しなくてもコマンド出力で合否が分かる検証。**

```markdown
- [ ] `shared/types.ts` の SiteRole 型を 'admin' → 'site_admin' に変更
- [ ] 🔬 `npx tsc --noEmit` → 型エラーなし
- [ ] 🔬 `npm test` → 全テストパス
```

期待結果まで書くことで「テスト確認しました」と曖昧に答える余地をなくします。画面を伴うPhaseでは、Playwright MCP でCSSを注入して**注目箇所を赤枠ハイライトしたスクリーンショット**を取得（バグ修正は Before/After ペア）。チェックの集約は `precheck` 1コマンドに寄せることを推奨しています。

## 別モデル・クロスレビュー — 実装とは違うモデルに差分を監査させる（任意）

3本柱はどの環境でも効く土台です。その上に、**実装したモデルとは別のモデルに実装差分をレビューさせる** 層を重ねられます。

同一モデルの自己レビューには構造的な限界があります。実装したモデルは、自分が「妥当」と判断した設計を、レビューでも同じ基準で「妥当」と通してしまう。DA（柱2①）は問い方でこれを突破しますが、それでも *モデル固有の* 死角は残る。実装がClaude系なら、Claude系が苦手な観点はDAを何周しても苦手なまま。**そこだけは、視点の違うモデルにしか埋められません。**

**どう組み込まれるか（Claudeが自動で回す）:**

- **Phase 0で要否を判定** — `必須`（API/DB/認可/入力検証/データ移行/複雑な状態遷移…）/ `推奨`（画面の大幅変更・3ファイル以上…）/ `不要`（文言修正・軽微な整理）を、理由付きでDDに記録
- **実装後・コミット前に自動実行** — 差分＋背景情報（DD目的・スコープ・設計意図・制約）を依頼書にまとめ、別モデルにヘッドレスで委譲。findings を優先で受け取る
- **修正は実装側（Claude）で** — 別モデルはレビュー専用でリポジトリを触らない（read-only）。主作業セッションとの同時編集の衝突が起きない

**何が拾えるか（実運用の一例・匿名化）:** メール差分同期の変更で、実装側のレビューを通過したコードに、別モデル(Codex)がデータ取りこぼしの経路を1件見つけました —

```text
🤖 別モデル・クロスレビュー（Codex, effort=high / read-only）
[P1] ページ上限に達した回は取り込み基準点を進めないこと — gas/src/Code.gs:320
     受信が50件上限を超えると先頭50件だけ処理するが、失敗0件なら基準点を実行開始時刻へ前進。
     残りの古いスレッドは次回の重複ウィンドウより前になり、二度と検索窓に入らず恒久的に取りこぼす。
     → ページングするか、上限到達時は基準点を進めないガードが必要（取りこぼし防止が主目的の変更と矛盾）
```

ハッピーパスは正しく動き、テストも通る。だが「受信が上限を超えた回だけ基準点が先へ進む」境界条件は、実装したモデルの自己レビューが最も見落としやすい死角です。視点の違うモデルが差分をDDの目的（取りこぼし防止）と突き合わせ、恒久的なデータ取りこぼしをコミット前に捕まえています。これが対策表の「同一モデルの盲点」への具体的な答えです。

**誇張しないための線引き:**

- **任意機能** — 3本柱の価値はこれ無しで完結します。これは土台の上への"上乗せ"であって、前提ではありません
- **環境ゲート** — 現状の実装は **Codex CLI（サブスク認証）** のヘッドレス実行（`codex-review.sh`）。**API課金なし**、可否は `--check`（トークン消費なし）で自動判定し、**未導入環境は自動スキップ**。導入していない読者が何かを失うことはありません
- **万能ではない** — 別モデルにも偽陽性はあり、実行時の問題は静的レビューでは捕まえきれません。テスト・E2E・実ユーザーと組み合わせて初めて信頼性が補完されます

> 実装上の名称は **Codexレビューゲート**。手順は [templates/guides.md](templates/guides.md) §Codexレビューゲート、スクリプトは [templates/scripts/codex-review.sh](templates/scripts/codex-review.sh)。

## ドキュメント体系: フローとストック

DDは「変更の記録（フロー）」です。完了したDDの内容は、アーカイブ時に「現在形（ストック）」へ反映されます。**この反映が `/dd archive` の必須チェックリストに組み込まれている**のが特徴です。

```
変更のフロー                          現在形のストック
─────────────                         ─────────────────────────────
doc/DD/DD-042_xxx.md   ──archive──▶   doc/spec/          仕様書（画面・機能の正典）
  ├ 受け入れ基準                       doc/engineering-patterns.md  gotcha・定石
  ├ DA批判レビュー記録                 doc/decisions.md   長寿命の意思決定（ADR）
  └ 機械検証エビデンス                 doc/DOC-MAP.md     全ドキュメントの索引
                                          ▲
                              scripts/doc-check.sh が孤児・リンク切れを機械検証
```

- **仕様書**（[テンプレート同梱](templates/screen-spec-template.md)）: 「この画面の今の仕様は？」にDD横断grepなしで答える層。アーカイブ時に同期チェック
- **engineering-patterns.md**: DAで見つかった「同根パターン」など、横断で効く罠と定石の昇格先
- **decisions.md**: 「なぜこの技術・この方式なのか」を1決定10行で逆引きできるADR
- **DOC-MAP.md + doc-check.sh**: ドキュメントの索引と、その索引が腐っていないことの機械検証

## 同梱ツールキット

| ツール | 何をするか |
|--------|-----------|
| `/dd` スキル | DD の作成・参照・一覧・検索・ログ追記・アーカイブ |
| `/setup` スキル | 外部プロジェクトへの対話的導入（パス整合性チェック付き） |
| **Lintヒント基盤** | Lintエラーに「規約ID + 修正方針」を埋め込み、**エラー自体がAIへの修正指示になる**。編集直後にフックで即フィードバック（[templates/lint/](templates/lint/)） |
| **コーディング基準書** | P01〜P22 を YES/NO で判定できる形式で定義。LLM固有アンチパターン（型システム迂回・スタブ残置等）を含む（[templates/coding-standards.md](templates/coding-standards.md)） |
| **別モデル・クロスレビュー**（任意 / 実装名: Codexレビューゲート） | 実装差分を別モデル（Codex）に監査させ、仕様/権限/テスト漏れを findings 優先で検出。サブスク認証CLIをヘッドレス実行し**API課金なし・read-only**。可否は `--check` で自動判定し未導入環境は自動スキップ（[templates/scripts/codex-review.sh](templates/scripts/codex-review.sh)） |
| **CLAUDE.md スニペット** | 50行程度に収めたリーンな CLAUDE.md の雛形（詳細は doc/ に置きポインタで参照） |
| **Playwright MCP 設定** | プロジェクトスコープの `.mcp.json`。プラグイン非対応環境（VSCode拡張等）でも動く |
| **フック** | 重要ファイル編集ガード / アーカイブ時のINDEX更新リマインダー / 編集直後Lint |
| **スクリプト** | DD-INDEX 全量再生成 / DOC-MAP 整合性チェック / **DD運用ヘルスチェック**（クローズ漏れ・滞留・ログ形骸化・DA雛形残置を静的分析。DD作成直後とアーカイブ前の即時セルフチェックにも使う） |
| **Modern Web Guidance 連携** | フロントエンド実装時に Chrome 公式のモダンWebガイドを `npx` で検索・取得する運用を規約に組み込み済み |

## 開発の流れ（5ステップ）

```
Step 1: DD作成     /dd new 機能名     ← 受け入れ基準・Phase構成を文書化
Step 2: 仕様確認   認識合わせ          ← ユーザーの「OK」を得てから進む
Step 3: 実装       Phase単位          ← 各Phaseに機械検証 + DA
Step 4: 検証       受け入れ基準照合    ← precheck（lint + テスト + doc-check）
Step 5: 完了       /dd archive        ← spec同期・知見昇格・INDEX再生成
```

コミットは `DD-{番号}: 概要` 形式。`git log --grep DD-042` がそのまま 仕様 → 決定 → コード のトレーサビリティになります。詳細は [doc/development-flow.md](doc/development-flow.md)。

## 設計思想

**なぜ「見落としチェック」ではなく Devil's Advocate か。** 「何を見落としているか？」と問うと、LLMは「特にありません」と答えがちです。「どこが壊れるか？」に変えるだけで、壊れるポイントを探すモードに切り替わります。

**なぜ件数ノルマを置かないか。** 「最低N件指摘せよ」はLLMに水増しを強制し、Critical判定の閾値を下げます。品質基準（読み返す価値）だけを置き、0件なら0件の根拠を示させる方が信頼できます。これは「細かい指示で縛るより、ゴールと制約を与える」という最新モデル向けプロンプティングの知見とも一致します。

**なぜ機械に寄せるか。** LLMの自己申告は信用せず、Lint・テスト・doc-check・スクリーンショットという「嘘のつけない出力」に検証を寄せます。逆に、機械化できた規約はプロンプトから削れる — Lintヒント基盤はその実装です。

**DAの限界も明示する。** 偽陽性は出る、実行時の問題は見つけにくい、ドメイン知識に依存する。「0件 = 安全」ではなく、テスト・E2E・実ユーザーフィードバックと組み合わせて初めて信頼性が補完されます。

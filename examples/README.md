## DDの種類と使い分け

### 機能実装DD

**用途**: 画面追加、機能追加、バグ修正など

**特徴**:
- スコープ（対象範囲・対象外）が明確
- 詳細設計（画面レイアウト、データフロー）を含む
- テストシナリオを含む
- フェーズ分割されたタスク一覧

**例**: ログイン機能の実装、一覧画面の追加、検索機能の追加

### 設計判断DD

**用途**: アーキテクチャ決定、規約策定、技術選定など

**特徴**:
- 検討内容が充実（複数の選択肢を比較）
- 決定事項が明確（なぜその選択をしたか）
- 成果物リスト（ドキュメント、設定ファイル）
- プロジェクト全体に影響

**例**: フレームワーク選定、コーディング規約策定、テスト戦略決定

## 実例から学べること

### 1. スコープの書き方

対象範囲と対象外を明確に分けることで、DDの境界が明確になります。

```markdown
## スコープ

### 本DDの対象範囲
| 機能 | 内容 |
|------|------|
| M-02 担当者変更画面 | 既存担当者の変更 |

### 対象外（後続DDで実施）
| 機能 | DD |
|------|----|
| M-04 新規追加 | 後続DD |
```

### 2. ログの書き方

日付ごとに、実施内容と判断理由を記録します。

```markdown
## ログ

### 2024-01-10
- DD作成
- 既存システム分析完了
- 方式Cを採用（理由：柔軟性が高い）

### 2024-01-11
- Phase 1 完了: ボタン追加
- Phase 2 完了: 画面実装
```

### 3. タスクのスキップ記録

不要になったタスクは `[-]` でマークし、理由を残します。

```markdown
## タスク一覧

- [x] 方式検討
- [x] 実装
- [-] 移行パターン対比表作成（実装経験で習熟、不要と判断）
```

## 自分のプロジェクトに適用する際のヒント

1. **最初は簡単なDDから始める** - 小さな機能追加やバグ修正で練習
2. **テンプレートを調整する** - プロジェクトに合ったセクションを追加
3. **ログを丁寧に書く** - 後から見て「なぜこうしたか」がわかるように
4. **完了したらアーカイブ** - 意思決定の履歴として価値がある
